Action space

The task is a single-particle cryo-EM reconstruction: raw electron-microscope movies go in, a 3-D density map comes out. The input movies are short stacks of frames, each stack recording the same field of frozen particles over the few seconds of exposure; they are sourced from EMPIAR, the Electron Microscopy Public Image Archive of raw cryo-EM data. The output is a 3-D density map, a voxel grid giving the reconstructed electron density of the particle. Finished maps are deposited in the EMDB (Electron Microscopy Data Bank) and the atomic models later built into them in the PDB (Protein Data Bank); those two archives are where the grader's reference structures come from. cryoSPARC is the software that performs the reconstruction. It is one of the two standard suites for single-particle cryo-EM (RELION is the other), and it covers the whole pipeline the task needs: motion correction, CTF estimation, particle picking, 2-D classification, ab-initio 3-D reconstruction, and refinement. That makes it the only tool in the loop here. The software on either side is out of scope: the microscope produces the movies, and once the map exists, atomic-model building and refinement (fitting a protein structure into the density) happen in Phenix and Coot, with ChimeraX or PyMOL for visualization.

Work in cryoSPARC is organized as a graph of jobs. Each job is one processing step, such as import, Patch Motion Correction, Patch CTF, a particle picker, 2-D Classification, Ab-initio Reconstruction, or Homogeneous Refinement. A job takes typed inputs (sets of micrographs, particles, or volumes), exposes parameters, and produces outputs that become the inputs of later jobs. A scientist normally assembles this graph in cryoSPARC's web interface: create a job, connect its inputs to an upstream job's outputs, set its parameters, queue it on a GPU, look at the result, and move on. A few steps are interactive. They cannot be set once and left alone, because they require looking at an intermediate result and making a judgment call: which 2-D class averages are real particles rather than junk, where to set the threshold that separates particles from contamination in a set of picks, and what level to contour a mask at. The agent drives this same machinery through code rather than the web interface. It connects to a live cryoSPARC instance over the official cryosparc-tools Python API, creates and queues jobs on the GPU cluster, reads their outputs back, and for the interactive steps requests a rendered image of the intermediate result and decides from it.

The agent begins from a fresh cryoSPARC workspace that contains only the raw input for this structure (the EMPIAR-10025 movies and their acquisition metadata: pixel size, voltage, dose) and a short brief stating the goal. It acts through exactly two tools:

  • run_python(code): the agent submits a string of Python. The harness runs it in one namespace on the cryoSPARC server that persists across calls, then returns whatever the code printed, the value of its last expression, and any error traceback. This is the only way the agent can do anything; it writes code that drives cryoSPARC through the cryosparc-tools API.
  • finish(summary): the agent calls this once a final map exists, to end the run and report the resolution it reached.

That namespace is pre-loaded with a connected cryoSPARC client (cs), the names of the workspace and the GPU lanes, the location of the raw data, and a set of inspection helpers that render an intermediate result as an image: a 2-D class montage, picks drawn on a micrograph, a slice through a volume, or a mask at several thresholds. The agent is not given ready-made functions for the processing steps or a list of the available job types. It queries the live instance for the job types and works out how to drive each one.

Everything the agent does, it does by writing code in run_python. With it the model can create a job and set its parameters, connect a job's inputs to the outputs of an earlier job, queue a job on a GPU and poll until it finishes, and read a finished job's numbers and output files. It can call an inspection helper to get an image of an intermediate result and look at it, then make the curation decisions that have no single correct answer: which 2-D classes to keep, where to put the particle-versus-junk threshold on a set of picks, what mask to apply, what symmetry to impose, and which refinement to run. When a job fails or a result looks wrong, the model reads the traceback or the metrics and submits revised code.

What the run produces, and what the grader scores, is the final sharpened 3-D density map together with the complete job tree that built it. The map is scored for its quality, and the tree of jobs, along with the decisions recorded in it, for the soundness of the procedure.

Task overview

The dataset the agent reconstructs, and the instructions and context it is given.

Data

Every core rollout runs on one dataset, the 20S proteasome (T20S), the standard specimen in the public cryoSPARC introductory tutorial. The raw data is EMPIAR-10025, and the rollouts use the tutorial's 20-movie subset of it. T20S is a roughly 700 kDa proteasome with high (D7) symmetry, which makes it small, rigid, and comparatively easy to reconstruct. The tutorial is cryoSPARC's step-by-step walkthrough from these movies to a ~3 Å map; we ran it by hand to produce the golden reference, and the same recipe is what the agent is given. The deposited answer key is EMD-6287 for the map and PDB 6BDF for the atomic model.

The benchmark is being extended to harder structures with a public deposit but no by-hand golden trajectory, each chosen to span a difficulty the tutorial specimen avoids. The full T20S dataset adds the rest of the movies to the one system that already has a golden trajectory, isolating the effect of scale. β-galactosidase, the standard RELION benchmark, is a larger, lower-symmetry particle on a trusted reference, a controlled step up from T20S. The CB1–Gi and MRGPRX2 GPCR complexes are small, flexible membrane proteins in detergent, where handedness, anisotropy, and a lower attainable resolution become the hard parts. The encapsulin is a large icosahedral assembly that probes the opposite extreme of very high symmetry. And the HIV Env BG505 trimer with a 44m Fab is the Scripps pilot system itself: flexible, low in symmetry, and the structure where the right processing decisions are genuinely uncertain rather than handed over by a recipe.

Agent inputs

Each rollout hands the model three things:

  • a system persona: a senior structural biologist working autonomously;
  • a task brief stating the goal and the acquisition facts the work needs (pixel size, voltage, dose, gain reference, and the molecule's symmetry and size);
  • reference material: the public cryoSPARC tutorial and the cryosparc-tools API notes.

It is given no pre-built job functions and no job catalog; it discovers the available job types from the live instance and decides the workflow itself.

Difficulty is set by how much of the recipe the brief includes, with the data and the tools held fixed. Handheld adds guidance distilled from the by-hand run on top of the tutorial: the job catalog, parameter values, and explicit decision hints. Tutorial gives exactly the public tutorial. Realistic gives a PI-style goal with the step-by-step recipe removed, so the model has to supply the workflow and every curation decision from its own expertise. Each step up the ladder removes scaffolding and moves more of the scientific judgment onto the model.

Rollouts

The benchmark is the set of rollouts below. Each rollout is one run of the reconstruction, by the agent or by a reference actor, on a stated dataset. They fall into six kinds. The tables list every rollout with a short description of what it is and how it was done; the rollout catalogue further down gives a summary card and, where available, a full report for each. Per-run grader scores live on the benchmark page.

  • Golden path. A human builds and runs the whole movies-to-map pipeline by hand, following the tutorial. It defines the golden trajectory and answer keys, and sets the quality ceiling the agent is measured against.
  • Performance floor. A minimal pipeline that makes random choices at the interactive steps, a no-skill lower bound that anchors the bottom of the score range.
  • Baseline (CryoWizard). A non-agentic automated pipeline run on the same data, a comparison point that is neither an expert nor an agent.
  • Agent rollouts. The model driving the full reconstruction itself, run at three levels of guidance: handheld, tutorial, and realistic.
  • Quality control (degraded data). Deliberately corrupted maps (wrong-hand mirror, added noise, half density) fed to the grader to confirm it separates good reconstructions from bad. Not a model run.
  • Realistic datasets. Reconstructions of structures other than the tutorial specimen, larger and harder and lower in symmetry, graded on the reference-free and deposit-anchored subscores.
