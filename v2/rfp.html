Scientific evaluation environment

Single-particle cryo-EM reconstruction — v0 evaluation-environment specification

Submitted in response to the RFP Evaluation Environments for Scientific Tasks (May 2026). This document specifies a v0 evaluation environment for single-particle cryo-electron microscopy (cryo-EM) reconstruction: the iterative, judgment-heavy pipeline that turns raw electron-microscope data into a 3D structure. It is written to be concrete enough to begin implementing, and it is instantiated on a public dataset that runs end to end today.

v0 specification single-particle cryo-EM reconstruction runnable on public data submitted June 2026

Note. The RFP prescribes execution-graded environments generically — named, weighted, computable subscores graded against held-out ground truth. Everything domain-specific below (the particular metrics, their weights, the action surfaces, and the accessions) is our instantiation of those generic requirements, not a restatement of them.

1. Task definition

What the agent does. The agent performs a complete single-particle cryo-EM reconstruction: from raw electron-microscope movie frames and their acquisition metadata, it drives the full processing pipeline and must deliver a refined 3D density map of the imaged macromolecule — the entire iterative loop a structural biologist runs, not an isolated step.

Input the agent receives.

  • Raw multi-frame movies for one dataset.
  • Acquisition metadata: pixel size, accelerating voltage, spherical aberration (Cs), total exposure dose, gain reference, detector type.
  • The point-group symmetry to impose at refinement and the particle-picker diameter(s) — given, not chosen.
  • A programmatic, no-GUI interface to a standard reconstruction package (cryoSPARC, via its Python API).
  • Depending on the configured guidance level: the public processing tutorial this environment tracks, and/or a job-type catalog with parameter values, and/or only a supervisor-style task brief plus the public API.
  • At each interactive step, a faithfully rendered, non-leaking observation (a class-average montage with per-class statistics; picks overlaid on a micrograph with the score histogram; map orthoslices and FSC curves) — images and aggregate statistics, with the raw per-particle arrays, the fitted cut, and the answer key withheld.

Actions the agent can take. The primitive action vocabulary is: create a job, run a job, read a job's results, and inspect-and-adjust an interactive job — and, at the most open level, write code directly against the package's Python API. The environment presents these at three action surfaces, crossed with how much guidance it supplies:

Action surfaceWhat the agent emitsGuidance range
Single decisionthe choice for one interactive step (kept-class indices, a threshold, an iso-level), executed by the environmentone isolated task per decision, with correct upstream state supplied
Constrained tool callsone call per step over the verb set above; builds and runs the whole graph, every decision inlinefully specified workflow → tutorial-only minimal catalog
Free-form codePython written against the package APIfull catalog + guidance → only a supervisor brief, the public tutorial, and the public API, with the agent discovering the job types itself

Artifact it must produce. A final 3D density map (sharpened, with its two half-maps), its global resolution from the gold-standard Fourier shell correlation (FSC), the refinement mask, and the realized job graph — job types, parameters, and intermediate outputs — that produced it.

Formally, the agent constructs and executes a job graph τ = (j₁,…,jₙ), each jᵢ a (type, parameters, input-edges) triple, yielding a final map M with half-maps; the environment returns a score g(M, τ) against held-out ground truth (§3).

What makes the task hard. The pipeline is a directed graph, not a linear script — several parameters are derived from a prior job's output rather than supplied as inputs — and within the otherwise-fixed protocol exactly five steps are genuine interactive judgments rather than fixed procedure:

import movies → motion correction → CTF estimation → (denoise) → particle picking
   → inspect/filter picks → extract → 2D classification → select picking templates
   → template-based picking → inspect/filter picks → extract → 2D classification
   → select refinement classes → ab-initio 3D (no symmetry) → 3D refinement (symmetry imposed)
   → mask generation + FSC validation → B-factor sharpening

The five judgments the agent owns: two particle-pick threshold cuts (after the first and second picking passes — the cross-correlation/power cut separating particles from junk); two 2D class-average selections (which classes become picking templates, and which feed 3D refinement); and one mask iso-level. The point-group symmetry and the particle diameter are given, not chosen.

2. Data

Primary dataset. EMPIAR-10025, the T20S proteasome — the 20-movie subset used by the standard public reconstruction tutorial (Titan Krios, 300 kV, K2 detector). Acquisition parameters (task-intrinsic): pixel size 0.6575 Å, voltage 300 kV, Cs 2.7 mm, dose 53 e/Å², gain reference norm-amibox05-0.mrc, symmetry D7.

Ground truth. The published, peer-reviewed reconstruction of this specimen, held out from the agent:

ReferenceWhat it isRole
Deposited atomic model (PDB 6BDF)the re-refined atomic model of the structureprimary structural answer key — did the agent build the right structure
Deposited density map (EMD-6287)the full-dataset 2.8 Å reconstruction + half-mapssupporting map-to-map cross-check
In-house expert reconstructionthe same 20-movie subset processed by hand to ≈3.15 Åthe same-subset resolution / particle-count reference and the procedure target values

Measurement model. The deposited map and model come from the published structure-determination pipeline for this specimen. We do not compare against numbers printed in the paper: every agreement metric is recomputed by us, under one fixed engine and procedure applied identically to every map — including the deposited reference itself.

Splits.

  • Held-out (in place): the deposited model and map, and all agreement metrics computed against them. The agent never reads the answer key.
  • Perturbed / resampled (in place): a degraded-map control suite — a wrong-hand (mirror) variant, a fewer-particles / half-density variant, and an added-noise variant — used to confirm the scorer drops on quality differences the resolution number cannot see.
  • External / transfer (proposed): a harder public EMPIAR dataset, and then the same structural workflow on an access-restricted dataset (§10). The pipeline is identical across splits, which is what makes transfer meaningful.

Note — resolution ceiling. Runs use a 20-movie subset, whose achievable ceiling is ≈3.1 Å. A run is never penalized for not reaching the 2.8 Å full-dataset figure: resolution bands are set against the same-subset reference, and structural agreement is normalized to the deposited reference low-pass-filtered to each run's own resolution.

3. Feedback signals and scoring

The environment returns named, computable subscores, each a number in [0,1] with a stated direction — never a qualitative judgment. They are organized into three weighted blocks — outcome, procedure, integrity — which are the three named signals the headline reduces to; the per-block subscores below are their computable decomposition. The signals an agent legitimately sees during interaction (its own half-map FSC, the package's plots, intermediate particle counts) are distinct from the held-out agreement metrics used only for final scoring (§4).

Reduction to a single number. A full trajectory reduces to a weighted sum of the named subscores — the RFP's "weighted combination of signals" — grouped into the three blocks above, so the headline is three named, weighted components (outcome / procedure / integrity), outcome-dominated, with the handedness gate applied multiplicatively to the outcome block. Provisional block weights are outcome 0.50 / procedure 0.35 / integrity 0.15.

4. Reward integrity

  • Visible to the agent during interaction: its own half-map FSC, the reconstruction package's plots, and intermediate particle counts — everything a working scientist legitimately sees.
  • Held out for final scoring: the deposited model and map, and every agreement metric computed against them.
  • Main degenerate strategy — "wishful masking" / over-fitting: a too-tight mask inflates the self-reported FSC while the underlying map disagrees with the truth. The defense is structural: the held-out map–model agreement scores such a map down regardless of how good its self-reported FSC looks, and the within-run mask-gap integrity signal flags the inflation directly.
  • State the agent must not read or alter: the held-out reference files, the grading code, and any cached answer key.
  • The core reward is rule-based and computable — no learned reward model and no LLM-as-judge, both of which are gameable. (A model-graded or image-based check could serve as a separate, optional quality signal, never as part of the core score.)

The degraded-map control suite (§2) confirms the held-out signals behave: a wrong-hand map is zeroed by the handedness gate, and a half-density map — which passes the handedness gate — is caught instead by the map–model block, while honest runs sit well above both.

5. Baseline and golden path

Golden path. The expert by-hand trajectory: a structural biologist driving the full pipeline to ≈3.15 Å on the subset, with the complete job graph and parameter choices recorded. This is both the reference trajectory procedure is graded against and the source of the procedure target values.

Baseline the agent should beat — a resolution-only scorer. Ranking runs by final resolution looks entirely reasonable in-sample, but it fails on exactly the signal that matters: every end-to-end run reaches ≈golden resolution (3.11–3.18 Å) regardless of decision quality, so a resolution-only metric rates them all equal and cannot rank the science. Beating this baseline — recovering the quality differences resolution conflates — is the environment's reason to exist. Correct procedure is itself part of the ground truth here, which is why the procedure block carries real weight. A rigid, fixed-heuristic automation of the same pipeline serves as a second, floor-level baseline.

6. Difficulty

The naive metric is saturated; the real quality signal is not. Every end-to-end run reaches ≈golden resolution, yet the procedure and integrity terms keep the sloppy runs visibly below the clean ones (0.75–0.77 vs 0.86 above). The cleanest evidence of non-triviality: a run that kept 29 picking templates (it should keep about two) and skipped both denoising and mask generation still posted the best nominal resolution (3.11 Å) — and the scorer ranked it below the clean runs anyway. The headroom lives in judgment, not in the final scalar.

On this single, well-behaved instance the held-out outcome block is itself near-saturated, so most of the current discrimination comes from the procedure block and from the perturbation and transfer splits. Pushing even clean-procedure runs meaningfully below ceiling on outcome is the role of harder instances — see the problem family (§7).

7. Problem family

Today this is a single fixed instance — valuable as a held-out eval — together with a within-instance perturbation family already in place (the degraded-map control suite: mirror / low-particle / noise).

For training value we propose a parameterized family of instances over three axes:

  • dataset — a handful of public EMPIAR single-particle sets spanning a range of symmetries, particle sizes, and difficulties;
  • pipeline entry stage — the agent enters at raw movies, at motion-corrected micrographs, or at pre-extracted particles;
  • perturbation — none, mirror, particle subsampling at set fractions, or added noise at set levels.

A generator script emits one manifest row per point in the product of these axes — a stable instance id mapping to its configuration — yielding the RFP's "manifest of problem instances generated by a script you write." The perturbation axis is already enumerated; the dataset and entry-stage axes are the proposed extension.

8. Metadata

  • Expected human completion time (one instance): roughly a half to a full working day for a trained user end to end, counting the interactive steps and queue/compute waits; the compute itself is GPU-hours.
  • Difficulty tier (relative to current models): medium — distinct from the data-access tiers in §10. Frontier models reach a golden-class map but make materially sub-optimal judgment calls, which is the gap this environment measures.
  • Grading approach: hybrid, programmatic-dominant — computable map-agreement metrics plus structured-field procedure checks; no LLM-judge in the core.

9. Four-to-six-week plan

A pilot that ends with the environment runnable end to end under a fixed compute budget. Weeks 1–4 already run as a working prototype on the public data; the schedule below is how we would sequence a clean delivery.

WeekDeliverable
1Stage the public data; run and record the expert golden path; reproduce it programmatically.
2Stand up the action surface — the agent drives the pipeline on a live instance, from single decisions to end-to-end.
3Collect and catalogue agent rollouts across the guidance/autonomy range.
4Implement the scorer — named weighted subscores with a held-out outcome — and validate its sensitivity on the degraded-map control suite.
5Wire in the baselines (the resolution-only scorer and the rigid pipeline); calibrate and freeze the weights.
6Package the containerized environment (setup + grading hooks), the instance manifest from a generator script, the data-fetch step, and stable train/test splits; confirm an end-to-end run under a fixed budget.

10. Data access

Runnable from day one on public data, with a tiered path to the encumbered data where the deepest value is:

  1. Public, day one (no agreement): EMPIAR-10025 and the deposited answer key (the model and map), all fetched from public archives. The entire v0 runs on this.
  2. Light credentialing: a harder public EMPIAR dataset, to extend the difficulty and transfer range.
  3. Under a data-use agreement: the non-public reconstruction pipeline data behind a published structure — raw micrographs, particle selections, intermediate maps, and the model-building/refinement logs, parameter choices, and provenance (~20 TB) — held by a partner research institute. The same structural workflow scales continuously from the public anchor to this dataset.

Shareable vs. encumbered. The public tier is shareable for this purpose immediately. The encumbered tier comes online under a data-use agreement now in negotiation with the partner institution; the public tier is fully sufficient to build, run, and evaluate the v0 environment in the meantime.

How this meets the evaluation criteria

  • Technical feasibility. Demonstrated rather than promised: the environment has been built and run end to end on obtainable public data, on a single cloud GPU instance, with computable metrics, an implemented scorer, and a real baseline.
  • Scientific merit. The task is the genuine iterative reconstruction loop a structural biologist runs, graded against a published, validated answer key — with a held-out outcome that confirms the right structure was built and a procedure layer that measures whether the science was done well, the difference a resolution number hides. A model that scores well here is more useful for the underlying research problem, not just better at the narrow metric.

What the pilot builds toward

Per the RFP's "what you will eventually build": a containerized environment with two hooks — a setup function returning the task prompt and materials, and a grading function returning named numeric subscores with weights — plus a manifest of problem instances from a generator script, a data-fetch step, a trivial baseline, and stable train/test splits. The scoring module is kept harness-agnostic so it survives integration into any evaluation framework.

Open questions for the pilot

  • Calibrating and freezing the block and within-block weights on the degraded-map control suite (they are provisional today).
  • Formalizing and enumerating the instance family {dataset × entry-stage × perturbation} via the generator script.
  • Characterizing per-decision variance: the pipeline is stochastic and runs are single-shot, so each interactive decision should be repeated several times to grade the distribution rather than a single draw.
  • Selecting the specific harder public dataset for the transfer split.
  • Capturing frozen observation snapshots so a fixed benchmark can replay without driving the live pipeline.
